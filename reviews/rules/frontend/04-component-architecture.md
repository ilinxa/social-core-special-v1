# 04 — Component Architecture Rules

Pass/fail criteria for each checklist item. Apply these when auditing the codebase.

---

## 4.1 Server vs Client Components

| ID | Rule | Verdict |
|----|------|---------|
| 4.1.1 | FAIL if components default to client-side rendering — files without "use client" must remain server components | PASS/FAIL |
| 4.1.2 | FAIL if "use client" is placed on a parent container or layout when only a child needs interactivity | PASS/FAIL |
| 4.1.3 | FAIL if any "use client" file imports server-only modules (database, fs, Node.js APIs, secret env vars) | PASS/FAIL |
| 4.1.4 | WARN if a component has "use client" but contains no hooks, event handlers, or browser API usage | PASS/WARN |
| 4.1.5 | FAIL if Providers.tsx does not compose all root-level client providers (QueryClientProvider, ThemeProvider, AuthInitializer, Toaster) | PASS/FAIL |
| 4.1.6 | WARN if hooks accessing window, localStorage, navigator, or IntersectionObserver lack "use client" in their file | PASS/WARN |
| 4.1.7 | FAIL if a client component imports a server-only module via direct import (not as children/props) | PASS/FAIL |

## 4.2 Component Declaration Style

| ID | Rule | Verdict |
|----|------|---------|
| 4.2.1 | FAIL if top-level component exports use arrow functions instead of named function declarations | PASS/FAIL |
| 4.2.2 | FAIL if the props interface/type is not declared immediately before the component function | PASS/FAIL |
| 4.2.3 | WARN if arrow function component exports exist — function declarations preferred for hoisting and stack traces | PASS/WARN |
| 4.2.4 | FAIL if non-Next.js convention files (anything other than page/layout/error/loading/not-found.tsx) use anonymous default exports | PASS/FAIL |
| 4.2.5 | WARN if component files do not follow imports → type → function → return JSX ordering | PASS/WARN |
| 4.2.6 | WARN if the exported component name does not match the filename (PascalCase or kebab-case mapping) | PASS/WARN |

## 4.3 shadcn/ui Primitive Usage

| ID | Rule | Verdict |
|----|------|---------|
| 4.3.1 | FAIL if components/ui/ contains manually created components not generated by `npx shadcn add` | PASS/FAIL |
| 4.3.2 | WARN if shadcn primitives in ui/ have been hand-modified beyond minor fixes (customization belongs in common/) | PASS/WARN |
| 4.3.3 | FAIL if project-specific composed components (FormField, PasswordInput, ConfirmActionDialog) are in ui/ instead of common/ | PASS/FAIL |
| 4.3.4 | WARN if composed components with multiple visual variants do not use CVA (class-variance-authority) | PASS/WARN |
| 4.3.5 | FAIL if shadcn component overrides use hardcoded hex/rgb colors instead of OKLCH design tokens | PASS/FAIL |
| 4.3.6 | WARN if new shadcn components were manually copied from website instead of installed via CLI | PASS/WARN |
| 4.3.7 | WARN if customizations in ui/ files prevent clean CLI updates of shadcn components | PASS/WARN |

## 4.4 Composition Patterns

| ID | Rule | Verdict |
|----|------|---------|
| 4.4.1 | WARN if complex components use large configuration objects instead of children/render props | PASS/WARN |
| 4.4.2 | WARN if the Radix asChild pattern is not used when a component needs to render as a different element | PASS/WARN |
| 4.4.3 | FAIL if props are drilled through more than 3 intermediate components that don't use them | PASS/FAIL |
| 4.4.4 | WARN if feature-scoped context is not used for shared data within a feature boundary (e.g. bconsole) | PASS/WARN |
| 4.4.5 | WARN if complex compound widgets (tabs, accordions, form builders) don't use the compound component pattern | PASS/WARN |
| 4.4.6 | WARN if slot-based composition (Card.Header, Card.Footer) is replaced by boolean props (showHeader, showFooter) | PASS/WARN |
| 4.4.7 | WARN if render callbacks/children-as-function lack explicit TypeScript types | PASS/WARN |

## 4.5 Permission-Aware Components

| ID | Rule | Verdict |
|----|------|---------|
| 4.5.1 | FAIL if permission-gated UI elements do not use the `<Can>` component | PASS/FAIL |
| 4.5.2 | FAIL if permissions are fetched separately instead of extracted from `_permissions` on GET detail responses | PASS/FAIL |
| 4.5.3 | FAIL if `_permissions` is injected on PATCH, POST, DELETE, or list responses | PASS/FAIL |
| 4.5.4 | FAIL if any component checks role names directly (role === "admin") instead of using boolean permission flags | PASS/FAIL |
| 4.5.5 | WARN if `<Can>` components do not accept a fallback prop for alternative content | PASS/WARN |
| 4.5.6 | FAIL if unauthorized UI elements are shown in a disabled/grayed-out state instead of hidden entirely | PASS/FAIL |

## 4.6 Forms & Controlled Components

| ID | Rule | Verdict |
|----|------|---------|
| 4.6.1 | FAIL if any form does not use react-hook-form with zodResolver for validation | PASS/FAIL |
| 4.6.2 | FAIL if forms use raw `<input>` instead of common components (FormField, FormTextarea, PasswordInput) | PASS/FAIL |
| 4.6.3 | WARN if validation schemas are not in lib/validations/ (shared) or co-located with their feature form | PASS/WARN |
| 4.6.4 | FAIL if controlled and uncontrolled form state are mixed in the same form | PASS/FAIL |
| 4.6.5 | FAIL if non-native inputs (Select, Combobox, DatePicker, TagInput) are not wrapped with Controller | PASS/FAIL |
| 4.6.6 | WARN if native text/email/password inputs use Controller instead of register() for no reason | PASS/WARN |

## 4.7 Dialog & Sheet Patterns

| ID | Rule | Verdict |
|----|------|---------|
| 4.7.1 | FAIL if modal dialogs use custom implementations instead of shadcn Dialog | PASS/FAIL |
| 4.7.2 | WARN if multi-step dialogs leak internal state (step progression, form state) to parent components | PASS/WARN |
| 4.7.3 | FAIL if dialog open/close state is not controlled by the parent via open + onOpenChange props | PASS/FAIL |
| 4.7.4 | WARN if dialogs or sheets cause body scroll lock issues on mobile (residual lock after unmount) | PASS/WARN |
| 4.7.5 | WARN if destructive actions do not use ConfirmActionDialog for confirmation | PASS/WARN |
| 4.7.6 | WARN if mobile navigation does not use Sheet (MobileMenuSheet) for slide-out panel | PASS/WARN |

## 4.8 Loading States

| ID | Rule | Verdict |
|----|------|---------|
| 4.8.1 | FAIL if loading states use spinner icons or "Loading..." text instead of Skeleton components | PASS/FAIL |
| 4.8.2 | WARN if data-dependent sections lack corresponding skeleton components | PASS/WARN |
| 4.8.3 | WARN if skeletons do not match eventual content's structure and dimensions (causing CLS) | PASS/WARN |
| 4.8.4 | FAIL if visible layout shift occurs when transitioning from skeleton to loaded content | PASS/FAIL |
| 4.8.5 | WARN if skeleton containers lack role="status" and aria-label="Loading" for accessibility | PASS/WARN |
| 4.8.6 | WARN if infinite scroll lists do not show skeleton cards at bottom during next-page fetch | PASS/WARN |

## 4.9 Error Display

| ID | Rule | Verdict |
|----|------|---------|
| 4.9.1 | FAIL if major feature areas lack error boundaries to prevent full-application crashes | PASS/FAIL |
| 4.9.2 | WARN if error displays do not use consistent Card + destructive variant styling | PASS/WARN |
| 4.9.3 | FAIL if error messages show raw error codes, stack traces, or JSON to end users | PASS/FAIL |
| 4.9.4 | WARN if error states lack a "Try again" / reset / retry button | PASS/WARN |
| 4.9.5 | WARN if empty states and error states are visually identical (should be distinct) | PASS/WARN |
| 4.9.6 | WARN if error display components lack test coverage | PASS/WARN |

## 4.10 Component File Size & Complexity

| ID | Rule | Verdict |
|----|------|---------|
| 4.10.1 | WARN if any component file exceeds 200 lines of code | PASS/WARN |
| 4.10.2 | WARN if multi-section UIs are not split into sub-components with a directory/index structure | PASS/WARN |
| 4.10.3 | WARN if a single file exports multiple unrelated components | PASS/WARN |
| 4.10.4 | WARN if components with significant state/logic do not extract that logic into custom hooks | PASS/WARN |
| 4.10.5 | WARN if JSX nesting exceeds 5 levels deep without extracting sub-components | PASS/WARN |
| 4.10.6 | WARN if pure utility functions are defined inside the component function body (re-created each render) | PASS/WARN |

## 4.11 Component Testing

| ID | Rule | Verdict |
|----|------|---------|
| 4.11.1 | FAIL if shared components in components/common/ lack co-located test files | PASS/FAIL |
| 4.11.2 | WARN if interactive feature components (buttons, forms, dialogs) lack test coverage | PASS/WARN |
| 4.11.3 | FAIL if component tests do not use renderWithProviders from test/utils.tsx | PASS/FAIL |
| 4.11.4 | WARN if tests only cover initial render without asserting user interaction and edge cases | PASS/WARN |
| 4.11.5 | FAIL if snapshot-only tests exist without supplementary behavioral assertions | PASS/FAIL |
| 4.11.6 | FAIL if tests do not mock external dependencies (API calls, router, stores) | PASS/FAIL |
| 4.11.7 | WARN if tests use getByTestId instead of accessible queries (getByRole, getByLabelText, getByText) | PASS/WARN |
